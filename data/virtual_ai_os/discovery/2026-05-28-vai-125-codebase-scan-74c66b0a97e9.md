## Resolution (VAI-125)

**Bug found and fixed** in `_apply_task_aging` (line ~1148):

The guard condition `int(task.priority - starvation_boost) < task.priority` diverged
from the actual priority calculation `max(0, task.priority - int(starvation_boost))`.

- Condition evaluates `int(5 - 0.7) = 4 < 5` → **true** (triggers requeue)
- Actual new priority: `5 - int(0.7) = 5 - 0 = 5` → **no change**

When `starvation_boost < 1.0` the task was requeued with identical priority every
aging tick, churning the queue and cancelling valid pending entries unnecessarily.

**Fix**: Compute `boosted_priority` once using the same formula as the PrioritizedTask
constructor, then gate the requeue on `boosted_priority < task.priority`.
The vague "hack" annotation was replaced with a clear explanation of the approach.
