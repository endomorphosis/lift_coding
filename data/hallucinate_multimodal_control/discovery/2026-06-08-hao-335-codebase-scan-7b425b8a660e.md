# HAO-335 Codebase Scan: Swallowed Exception in fix_s3_backend.py:698

## Finding

**File:** `external/ipfs_kit/archive/applied_patches/fix_s3_backend.py`  
**Line:** 698  
**Method:** `get_metadata()` in `S3Backend`

### Issue

A bare `except Exception: pass` clause silently swallowed any exception raised when
reading cached metadata from disk. This made it impossible to diagnose cache corruption
or permission errors that triggered the S3 fallback path.

```python
# BEFORE (line 698) – exception swallowed with no diagnostic output
except Exception:
    # If anything goes wrong with cache, fall back to S3
    pass
```

### Root Cause

The cache-read block in `get_metadata()` did not capture the exception variable,
so no log record was emitted. Every other similar cache-fallback site in the same
file (lines 543, 633) correctly logs via `logger.warning(...)`.

### Fix Applied

```python
# AFTER – exception captured and logged before falling back to S3
except Exception as cache_error:
    logger.warning(f"Error reading metadata from cache: {str(cache_error)}")
    # Fall back to S3
```

## Impact

- Operators can now see cache-read failures in logs rather than silently falling
  back to S3 on every request when the cache is broken.
- No behaviour change for the happy path.
