## Finding

The `backup_file()` function caught all exceptions, logged the error, and returned
`None` — a classic swallowed-exception pattern:

```python
# Before fix (lines 55-57)
except Exception as e:
    logger.error(f"Failed to back up {file_path}: {e}")
    return None
```

### Why this is a bug

All three callers (`update_storacha_kit`, `update_imports_in_file`,
`update_mcp_extension`) ignored the `None` return value.  If the backup failed
silently, the script would continue to overwrite the original file without a
working backup, making recovery impossible.

## Fix Applied

Re-raise the caught exception after logging so callers cannot silently bypass a
backup failure.  Each caller already wraps its work in a `try/except` block that
returns `False` on error, so the exception propagates correctly and the caller
aborts without destroying the original file.

```python
# After fix
except Exception as e:
    logger.error(f"Failed to back up {file_path}: {e}")
    raise
```

The docstring was also updated to document the `Raises` contract.
