## Resolution

Fixed in worktree `hao-375-attempt-1-1780998273`.

The broad `except Exception as e:` blocks in `find_providers()` (lines 1636 and 1664)
were catching all exceptions — including unexpected ones — and only logging them at
DEBUG level, effectively swallowing them.

### Fix applied

Split exception handling into two tiers:
1. **Expected parse failures** (`ValueError`, `TypeError`) — kept at DEBUG level, as these
   are normal when the request body is not JSON or form-encoded.
2. **Unexpected errors** (all other `Exception` subclasses) — escalated to WARNING level
   so they are visible in production logs.

This preserves the graceful fallback behavior (try JSON → try form → use default)
while ensuring that unexpected errors (e.g., `RuntimeError` from Starlette if body is
read twice, or I/O errors) are not silently discarded.
